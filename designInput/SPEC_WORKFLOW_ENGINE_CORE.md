# MES v2 - Workflow Engine Core Specification

## Purpose

Define a lightweight, reusable workflow kernel for MES v2 that can support multiple business processes (Hold Tags, NCRs, and future modules) without introducing full BPM platform complexity.

This spec standardizes:

- workflow definitions and runtime instances
- reusable work items (todo/action queue)
- reusable notification rules
- approvals and decisions
- audit/event model

## Design Principles

- Keep logic simple and explicit (no BPMN parser, no visual designer required).
- Use configuration for process flow and code for domain validations.
- Keep controller endpoints thin; workflow transitions execute in services.
- Record all transitions and decisions for traceability.
- Reuse shared notification and work-item infrastructure across process types.

## Scope

### In Scope (Core v1)

- Configurable step-based workflows.
- Deterministic transition rules.
- Work item generation/assignment/completion.
- Notification trigger rules and clear conditions.
- Step-level approvals (`None`, `AnyOne`, `All`).
- Audit/event history for all state transitions.

### Out of Scope (Core v1)

- Full BPMN/DMN modeling and visual designer.
- Arbitrary scripting language in process definitions.
- Cross-system orchestration and compensation semantics.
- Human task delegation chains and vacation substitution policies.

## Core Concepts

### WorkflowDefinition

Template describing a process type and version.

Required fields:

- `Id`
- `WorkflowType` (e.g., `HoldTag`, `Ncr`)
- `Version`
- `IsActive`
- `StartStepCode`

### WorkflowStepDefinition

Ordered step definition belonging to one `WorkflowDefinition`.

Required fields:

- `StepCode` (unique per definition)
- `StepName`
- `Sequence`
- `RequiredFields` (field keys validated by domain service)
- `RequiredChecklistTemplateIds` (optional list)
- `ApprovalMode` (`None`, `AnyOne`, `All`)
- `ApprovalAssignments` (users and/or roles)
- `AllowReject` (bool)
- `OnApproveNextStepCode` (nullable, null means terminal complete)
- `OnRejectTargetStepCode` (nullable)

### WorkflowInstance

Runtime process record attached to a domain entity.

Required fields:

- `Id`
- `WorkflowDefinitionId`
- `WorkflowType`
- `WorkflowDefinitionVersion`
- `EntityType` (e.g., `HoldTag`, `Ncr`)
- `EntityId`
- `Status` (`Draft`, `InProgress`, `PendingApproval`, `Rejected`, `Completed`, `Voided`)
- `CurrentStepCode`
- `StartedAtUtc`
- `CompletedAtUtc` (nullable)

### WorkflowStepInstance

Runtime execution record for each visited step.

Required fields:

- `Id`
- `WorkflowInstanceId`
- `StepCode`
- `Status` (`NotStarted`, `InProgress`, `PendingApproval`, `Approved`, `Rejected`, `Completed`, `Skipped`)
- `StartedAtUtc`
- `EndedAtUtc` (nullable)
- `CompletedByUserId` (nullable)

### WorkItem (Reusable Todo)

Actionable task generated by workflow events.

Required fields:

- `Id`
- `WorkflowInstanceId`
- `EntityType`
- `EntityId`
- `WorkItemType` (e.g., `Review`, `Approve`, `ProvideInfo`, `CompleteChecklist`)
- `Title`
- `Instructions` (optional)
- `AssignedUserId` (nullable)
- `AssignedRoleId` (nullable)
- `Status` (`Open`, `InProgress`, `Completed`, `Cancelled`)
- `Priority` (`Low`, `Normal`, `High`, `Critical`)
- `DueAtUtc` (nullable)
- `CreatedAtUtc`
- `CompletedAtUtc` (nullable)
- `CompletedByUserId` (nullable)

Rules:

- Exactly one of `AssignedUserId` or `AssignedRoleId` must be set at create time.
- Work items are immutable after `Completed` or `Cancelled`.
- Work item completion can trigger workflow transitions when configured.

### NotificationRule

Reusable rule defining who gets notified and when.

Required fields:

- `Id`
- `WorkflowType`
- `TriggerEvent` (e.g., `Created`, `StepEntered`, `SubmittedForApproval`, `Rejected`, `Completed`)
- `RecipientMode` (`Users`, `Roles`, `Resolvers`)
- `RecipientConfig` (users/roles or resolver key)
- `TemplateKey`
- `ClearPolicy` (`None`, `OnEntityComplete`, `OnStepExit`, `Manual`)

### WorkflowEvent (Audit/Event Log)

Immutable event stream for audit and diagnostics.

Required fields:

- `Id`
- `WorkflowInstanceId`
- `EventType`
- `EventAtUtc`
- `ActorUserId` (nullable for system events)
- `PayloadJson`

## State and Transition Model

- Workflow starts at `StartStepCode`.
- Entering a step creates or activates its `WorkflowStepInstance`.
- If `ApprovalMode = None`, completion can proceed directly after validation.
- If approval is required:
  - create approval work items
  - set workflow status to `PendingApproval`
  - collect approval decisions according to step mode
- On approve:
  - advance to `OnApproveNextStepCode`
  - if null, mark workflow `Completed`
- On reject:
  - require comments
  - if `OnRejectTargetStepCode` is configured:
    - mark current step `Rejected`
    - set workflow `Status = Rejected`
    - transition `CurrentStepCode` to `OnRejectTargetStepCode`
    - re-enter target step as active work according to step rules
  - if `OnRejectTargetStepCode` is null:
    - mark current step `Rejected`
    - set workflow `Status = Rejected`
    - keep `CurrentStepCode` on the rejected step (no implicit advance)
    - workflow is non-terminal and recoverable only via explicit domain/workflow action (for example, resubmit/rework action mapped to a valid transition)
    - if no explicit recovery transition is configured, further progression is blocked by design

## Approval Semantics

- `None`: no approval required.
- `AnyOne`: first approval completes the approval gate.
- `All`: all resolved approvers must approve.
- Rejection behavior:
  - step must have `AllowReject = true`
  - comments required
  - outstanding approval work items are cancelled

## Validation and Domain Ownership

- Core engine validates workflow structure and transition legality.
- Domain services validate business-specific rules and required data values.
- Transition only succeeds when both core and domain validations pass.

## Notifications and Work Items

- Notifications are event-driven from `WorkflowEvent`.
- Work items are generated at key lifecycle points (step entry, approval requests, rework requests).
- Notification delivery channel(s) are implementation-specific (in-app first; others optional later).

## Versioning and Migration Rules

- New process behavior creates a new `WorkflowDefinitionVersion`.
- Existing `WorkflowInstance` records continue using the version they started with.
- Definitions are append-only by version; avoid in-place mutation of active versions.

## API Boundary (Conceptual)

- `StartWorkflow(entityType, entityId, workflowType)`
- `AdvanceStep(workflowInstanceId, actionCode, actorUserId, payload)`
- `ApproveStep(workflowInstanceId, stepCode, actorUserId, comments?)`
- `RejectStep(workflowInstanceId, stepCode, actorUserId, comments)`
- `GetOpenWorkItems(userId, roleIds)`
- `CompleteWorkItem(workItemId, actorUserId, payload?)`

## Security and Access

- Access to transition actions is enforced by domain authorization + workflow assignment context.
- Only assigned approvers can approve/reject approval work items.
- Audit/event data is append-only.

## Non-Functional Requirements

- All timestamps stored in UTC.
- Transition operations are atomic and transactional.
- Event and work-item writes use outbox/eventing pattern for reliable delivery.
- Idempotency required for externally retried commands.

## Acceptance Criteria

1. A workflow definition can represent at least Hold Tag and NCR process shapes.
2. Workflow instances preserve definition version used at start.
3. Step transitions are blocked on invalid state/action.
4. Approval modes `None`, `AnyOne`, and `All` function correctly.
5. Work items are generated and queryable by user/role assignment.
6. Notification rules trigger on configured workflow events.
7. Rejection requires comments and records an auditable event.
8. Terminal workflows are immutable except read-only queries.
